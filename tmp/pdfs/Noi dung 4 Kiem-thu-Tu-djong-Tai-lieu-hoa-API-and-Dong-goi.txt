Viết Test Scripts trong Postman
Postman sử dụng JavaScript để viết test assertions. Các script này chạy sau khi nhận response từ server.
Test 1: Kiểm tra Status Code
pm.test("Status code is 200", function () {
pm.response.to.have.status(200);
});
Test 2: Kiểm tra Response Body
pm.test("Response contains token", function () {
var jsonData = pm.response.json();
pm.expect(jsonData.token).to.not.be.null;
pm.expect(jsonData.token).to.be.a('string');
});
Test 3: Lưu biến cho request tiếp theo
// Lưu token để dùng cho API khác
var jsonData = pm.response.json();
pm.environment.set("auth_token", jsonData.token);

Vấn đề kinh điển: N+1 Query Problem
Đây là lỗi phổ biến nhất gây chậm API. Xảy ra khi code thực hiện 1 query lấy danh sách, sau đó N queries riêng lẻ để lấy thông tin chi tiết của từng item.
Ví dụ: Hiển thị danh sách học viên kèm thông tin khóa học đã mua
// BAD: N+1 Query
SELECT * FROM users; // 1 query: lấy 100 users
// Sau đó loop qua từng user:
SELECT * FROM enrollments WHERE user_id = 1; // Query #2
SELECT * FROM enrollments WHERE user_id = 2; // Query #3
...
SELECT * FROM enrollments WHERE user_id = 100; // Query #101
// Tổng cộng: 1 + 100 = 101 queries!
So sánh: Before vs After Eager Loading
Before: N+1 Queries After: Eager Loading
// 101 queries // Chỉ 1 query với JOIN
SELECT * FROM users; SELECT users.*, enrollments.*
for each user: FROM users
SELECT * FROM enrollments LEFT JOIN enrollments
WHERE user_id = ? ON users.id = enrollments.user_id

Eager Loading trong thực tế
Hầu hết các ORM (Object-Relational Mapping) framework đều hỗ trợ Eager Loading.
Sequelize (Node.js)
const users = await User.findAll({
include: [{
model: Enrollment,
include: [Course] // Nested include
}]
});
// 1 query với JOIN thay vì N+1 queries
Laravel Eloquent (PHP)
$users = User::with('enrollments.course')->get();
// Tự động eager load relationships
Best Practice: Luôn sử dụng include hoặc with khi cần truy vấn dữ liệu liên quan. Bật query logging để phát hiện N+1 trong development.

Docker & Containerization
Đóng gói ứng dụng thành Container
Vấn đề: "Works on My Machine"
Dev: "Code chạy ngon lành trên máy tôi, không biết sao lên server lại lỗi?"
Ops: "Máy bạn Node.js 18, server đang dùng Node.js 14. MySQL của bạn 8.0, server là
5.7..."
Vấn đề môi trường không đồng nhất (environment inconsistency) gây lãng phí hàng giờ
debug. Mỗi developer có cấu hình khác nhau, server production lại khác nữa. Docker sinh
ra để giải quyết vấn đề này bằng cách đóng gói toàn bộ môi trường chạy vào một "thùng
container".
Container vs Virtual Machine
Virtual Machine (Cách cũ) Container (Docker)
• Mỗi VM chạy một OS riêng (Ubuntu, Windows...) • Chia sẻ kernel của host OS
• Nặng (vài GB), khởi động chậm (phút) • Nhẹ (vài MB), khởi động cực nhanh (giây)
• Tốn nhiều tài nguyên: CPU, RAM, Disk • Hiệu quả: Chạy hàng chục containers trên 1 server
Container cô lập về filesystem, network, process nhưng dùng chung kernel. Giống như các căn hộ trong một tòa nhà (chung nền móng) thay vì xây nhiều ngôi nhà riêng biệt
(VM).
Khái niệm cốt lõi: Image & Container
Docker Image Docker Container
Khuôn đúc (Template) Bánh đúc (Instance)
• File read-only chứa OS, code, dependencies • Runtime instance của Image, đang chạy
• Build một lần, dùng mãi mãi • Có thể start/stop/restart nhiều lần
• Lưu trữ trên Docker Hub hoặc Registry • Mỗi container có filesystem riêng
• Ví dụ: node:18-alpine, mysql:8.0 • Ví dụ: codelearning-api, codelearning-db
Tương tự như Class vs Object trong lập trình: Image là class, Container là object được khởi tạo từ class đó.
Dockerfile: Công thức nấu ăn
Dockerfile là text file chứa các lệnh để build Docker Image. Mỗi dòng lệnh tạo ra một "layer" trong image.
Dockerfile cho Backend Node.js (CodeLearning)
# Bước 1: Chọn base image
FROM node:18-alpine
# Bước 2: Tạo thư mục làm việc
WORKDIR /app
# Bước 3: Copy package.json và cài dependencies
COPY package*.json ./
RUN npm install --production
# Bước 4: Copy toàn bộ source code
COPY . .
# Bước 5: Expose port
EXPOSE 3000
# Bước 6: Lệnh chạy ứng dụng
CMD ["node", "server.js"]

Phân tích docker-compose.yml
Services
1
Định nghĩa các containers cần chạy. Mỗi service = 1 container. Có thể build từ Dockerfile (build) hoặc pull từ registry (image).
Ports
2
Map port từ host vào container theo format "host:container". Ví dụ: 3000:3000 nghĩa là truy cập localhost:3000 sẽ forward vào port 3000 bên trong
container.
Environment
3
Truyền biến môi trường vào container. Backend dùng DB_HOST=database để kết nối MySQL (tên service = hostname trong network).
Depends_on
4
Xác định thứ tự khởi động. Backend phải đợi database và cache start xong mới chạy. Tránh lỗi connection refused.
Volumes
5
Lưu trữ persistent data. MySQL data được mount vào volume db-data, không mất khi container restart.
Networks
6
Tạo mạng riêng cho các containers giao tiếp. Cô lập với containers khác trên cùng host.
